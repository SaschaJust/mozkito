8 joda-time 1.5+ issues with roundFloor and DST - ID: 1887104
Last Update: Comment added ( daliboz )

On versions 1.5+, using roundFloorCopy on one of the ambiguous times of the Fall DST change may cause a DateTime with the wrong hour to be returned. Example:

DateTime fallAmbig = new DateTime("2007-11-04T01:34:59-06:00");
DateTime americaHour = fallAmbig.minuteOfHour().roundFloorCopy();
System.out.println("New: " + americaHour.withZone(DateTimeZone.forID("America/Chicago")));

Produces: 2007-11-04T01:34:00.000-05:00

instead of 2007-11-04T01:34:00.000-06:00


Tested with java 1.6 update 4, windows XP SP 2

Jenni ( daliboz ) - 2008-02-05 08:24:58 PST

I noticed that when you use the DateTime.Property#roundFloorCopy on the

is less than an hour, the behavior has changed for the 1.5 series due to
the svn revision 1323 to fix this. Post-svn revision 1323, the behavior is
the same as it was on 1.4, and it's not documented what "should" happen
when something like this is requested. So I'm not sure if 1.5 - 1.5.2 was
'right' or if 1.4 was 'right.' What are your thoughts?

On 1.5 - pre-svn revision 1323:
Original: 2007-10-28T02:31:01.000+11:00
New: 2007-10-28T02:30:00.000+11:00

On 1.4 and post-svn revision 1323:
Original: 2007-10-28T02:31:01.000+11:00
New: 2007-10-28T01:30:00.000+10:30

Code to reproduce:
DateTime aussieSpring = new
DateTime("2007-10-28T02:31:01+11:00").withZone(DateTimeZone.forID("Australia/Lord_Howe"));

System.out.println("Rounding down hour of day:
Australia/Lord_Howe");
DateTime australiaMinute =
aussieSpring.hourOfDay().roundFloorCopy();
System.out.println("Original: " +
aussieSpring.withZone(DateTimeZone.forID("Australia/Lord_Howe")));
System.out.println("New: " +
australiaMinute.withZone(DateTimeZone.forID("Australia/Lord_Howe")));
